## 상세 테스트 내용

### UC-01 회원가입
- **입력:** 이름(테스트유저), 이메일(e2e_test_20260515@test.com), 비밀번호(8자 이상)
- **결과:** 가입 즉시 `/todos` 메인 화면으로 이동, 로그인 상태 유지 확인
- **검증:** 토큰이 Zustand 메모리에 저장되어 페이지 리로드 없이 인증 상태 유지

### UC-02 로그인
- **입력:** 동일 이메일/비밀번호
- **결과:** `/todos` 메인 화면으로 이동
- **부수 검증:** 페이지 직접 URL 이동(`window.location`) 시 Zustand 메모리가 초기화되어 로그인 화면으로 리다이렉트 — 명세(메모리 저장) 기반 정상 동작

### UC-03 사용자 정보 수정
- **수정 내용:** 이름 "테스트유저" → "E2E테스터"
- **결과:** 수정 후 사이드바 이름 표시 갱신 확인(로그인 재시 E2E테스터로 표시)

### UC-04 할일 등록
- **등록 항목:** "결제 모듈 버그 수정"(업무, 2026-05-15), "신제품 런칭 SNS 콘텐츠 기획"(개인, 2026-05-10~14), "2라인 생산량 점검"(업무, 2026-05-15)
- **결과:** 3건 모두 정상 등록, 목록에 즉시 반영

### UC-05 할일 수정
- **수정 대상:** "신제품 런칭 SNS 콘텐츠 기획" 종료예정일 2026-05-14 → 2026-05-19
- **결과:** 수정 모달이 기존 데이터로 채워진 상태로 열림, 저장 후 목록 갱신 확인

### UC-06 할일 삭제
- **삭제 대상:** "2라인 생산량 점검"
- **흐름:** 삭제 버튼 → 확인 다이얼로그("이 할일을 삭제하시겠습니까?") → 확인 클릭
- **결과:** 목록에서 즉시 제거

### UC-07 완료 처리 토글
- **검증 1 (미완료→완료):** "결제 모듈 버그 수정" 체크박스 클릭 → `checked` 상태로 전환
- **검증 2 (완료→미완료):** 재클릭 → 비체크 상태로 복원
- **결과:** 양방향 토글 모두 정상 동작

### UC-08 할일 목록 필터링
- **필터 1 (카테고리):** "업무" 선택 → 업무 카테고리 2건만 표시 (개인 카테고리 제외 확인)
- **필터 2 (AND 조합):** "업무" + "미완료" 조합 → 완료 처리된 "결제 모듈 버그 수정" 제외, 1건만 표시
- **결과:** AND 조합 필터링 정상 동작

### UC-09 카테고리 추가
- **입력:** "마케팅"
- **결과:** 사용자 정의 카테고리 섹션에 추가, 수정·삭제 버튼 활성화 확인
- **부수 검증:** 기본 카테고리(업무/개인/기타)의 수정·삭제 버튼은 `disabled` 상태로 UI에서 사전 차단 확인

### UC-10 카테고리 삭제
- **삭제 대상:** "마케팅팀" (수정 후)
- **흐름:** 삭제 버튼 → 확인 다이얼로그("이 카테고리를 삭제하시겠습니까?") → 확인 클릭
- **결과:** "추가된 카테고리가 없습니다" 안내 문구 표시

### UC-11 카테고리 수정
- **수정 내용:** "마케팅" → "마케팅팀" (인라인 편집 모드)
- **결과:** 수정 버튼 클릭 시 인라인 편집 폼으로 전환, 저장 후 목록에 변경된 이름 반영

### 로그아웃
- **흐름:** 사이드바 로그아웃 버튼 클릭
- **결과:** Zustand 스토어 초기화 → `/login` 즉시 이동

### 회원 탈퇴
- **흐름:** 설정 → "회원 탈퇴" 버튼 → 확인 다이얼로그 → 확인
- **결과:** `/login`으로 이동, 탈퇴된 계정으로 재로그인 시도 시 인증 실패(로그인 화면 유지) — 계정 완전 삭제 확인

## 발견된 이슈

### 이슈 1 (경미) — 사이드바 뷰포트 문제
- **현상:** 기본 뷰포트(1280×720 이하)에서 사이드바가 뷰포트 밖에 위치해 직접 클릭 불가
- **영향:** 테스트 시 메뉴 열기(☰) 버튼을 먼저 클릭해야 접근 가능
- **재현 조건:** 반응형 레이아웃에서 사이드바가 오버레이 방식으로 전환될 때

### 이슈 2 (정보) — Zustand 메모리 저장 동작 확인
- **현상:** 브라우저 URL 직접 입력(풀 페이지 로드) 시 토큰이 초기화되어 로그인 화면으로 리다이렉트
- **판정:** 명세(UC 정상 흐름, Access Token 재발급 예외 흐름) 기반 의도된 동작
- **사용자 영향:** 새로고침 시 재로그인 필요 — 명세에 명시된 내용

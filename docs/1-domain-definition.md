## 1. 프로젝트 개요

| 항목 | 내용 |
|------|------|
| 프로젝트명 | TodolistApp |
| 목적 | 인증된 사용자 단위로 할일을 안전하게 저장·관리하는 개인 할일 관리 애플리케이션 |
| 핵심 문제 | 사용자 인증 기반으로 개인 데이터를 격리하고, 카테고리 분류 및 다중 필터링을 지원하는 할일 관리 수단의 부재 |

## 2. 도메인 모델

### 2.1 User (사용자)

| 속성 | 타입 | NULL 허용 | 제약 조건 | 설명 |
|------|------|-----------|-----------|------|
| id | UUID | NOT NULL | PK, auto-generate | 사용자 고유 식별자 |
| email | String | NOT NULL | UNIQUE, 최대 254자, RFC 5321 형식 | 로그인 아이디 |
| password | String | NOT NULL | 최소 8자, bcrypt 해시 저장 | 암호화된 비밀번호 |
| name | String | NOT NULL | 최대 50자 | 사용자 이름 |
| createdAt | DateTime | NOT NULL | auto-set on insert | 가입일시 |
| updatedAt | DateTime | NOT NULL | auto-set on update | 정보 수정일시 |

### 2.2 Todo (할일)

| 속성 | 타입 | NULL 허용 | 제약 조건 | 설명 |
|------|------|-----------|-----------|------|
| id | UUID | NOT NULL | PK, auto-generate | 할일 고유 식별자 |
| userId | UUID | NOT NULL | FK → User.id | 소유 사용자 |
| categoryId | UUID | NOT NULL | FK → Category.id | 소속 카테고리 |
| title | String | NOT NULL | 최대 200자 | 할일 제목 |
| description | String | NULL 허용 | 최대 2000자 | 설명 정보 |
| startDate | Date | NOT NULL | — | 시작일 |
| dueDate | Date | NOT NULL | dueDate >= startDate | 종료 예정일 |
| isCompleted | Boolean | NOT NULL | 기본값: false | 완료 여부 |
| createdAt | DateTime | NOT NULL | auto-set on insert | 등록일시 |
| updatedAt | DateTime | NOT NULL | auto-set on update | 수정일시 |

### 2.3 Category (카테고리)

| 속성 | 타입 | NULL 허용 | 제약 조건 | 설명 |
|------|------|-----------|-----------|------|
| id | UUID | NOT NULL | PK, auto-generate | 카테고리 고유 식별자 |
| userId | UUID | NULL 허용 | FK → User.id; NULL이면 기본 카테고리 | 소유 사용자 |
| name | String | NOT NULL | 최대 50자, 동일 사용자 내 중복 불가 | 카테고리명 |
| isDefault | Boolean | NOT NULL | 기본값: false | 기본 카테고리 여부 |
| createdAt | DateTime | NOT NULL | auto-set on insert | 생성일시 |

> **기본 카테고리 구현 방식:** DB 시딩 방식으로 제공한다. 서버 기동 시 `업무`, `개인`, `기타` 3개 레코드를 `userId = NULL, isDefault = true`로 seed한다. 코드 상수가 아닌 DB 레코드로 관리하여 향후 관리자 추가가 가능하도록 한다.

### 2.4 도메인 관계

```
User 1 ──── N Todo
User 1 ──── N Category (사용자 정의)
Category 1 ──── N Todo
```

## 3. 비즈니스 규칙

### 3.1 사용자 (User)

| 식별번호 | 규칙 |
|----------|------|
| BR-3.1.1 | 이메일은 시스템 전체에서 고유해야 한다. |
| BR-3.1.2 | 회원가입 후 JWT 토큰이 발급되어야 앱 기능을 사용할 수 있다. 별도 이메일 인증 단계는 없으며, 로그인 성공 시 Access Token(만료: 1시간)과 Refresh Token(만료: 7일)을 응답 바디로 반환한다. 두 토큰은 클라이언트 Zustand 스토어(메모리)에만 저장하며 Cookie·localStorage는 사용하지 않는다. |
| BR-3.1.3 | 사용자는 자신의 정보(이름, 비밀번호)만 수정할 수 있다. |

### 3.2 할일 (Todo)

| 식별번호 | 규칙 |
|----------|------|
| BR-3.2.1 | 할일은 반드시 인증된 사용자에게 귀속된다. |
| BR-3.2.2 | 사용자는 자신이 소유한 할일만 조회·수정·삭제할 수 있다. |
| BR-3.2.3 | 할일 등록 시 카테고리는 필수이며, 기본 카테고리 또는 사용자 정의 카테고리 중 선택한다. |
| BR-3.2.4 | `dueDate`는 `startDate`와 같거나 이후 날짜여야 한다. |
| BR-3.2.5 | 완료 처리된 할일도 삭제 전까지 데이터를 보존한다. |

### 3.3 카테고리 (Category)

| 식별번호 | 규칙 |
|----------|------|
| BR-3.3.1 | 기본 카테고리는 모든 사용자에게 공통 제공되며 수정·삭제할 수 없다. |
| BR-3.3.2 | 사용자 정의 카테고리는 해당 사용자만 사용할 수 있다. |
| BR-3.3.3 | 카테고리 삭제 시 해당 카테고리에 속한 할일이 존재하면 삭제할 수 없다. |

## 4. 유비쿼터스 언어

| 용어 | 정의 |
|------|------|
| 사용자 (User) | 회원가입 후 JWT 토큰을 발급받은 앱 이용자 |
| 할일 (Todo) | 사용자가 등록한 수행 예정 작업 단위 |
| 카테고리 (Category) | 할일을 분류하는 그룹 단위 |
| 기본 카테고리 | 시스템이 DB 시딩으로 제공하는 공통 카테고리 (업무, 개인, 기타) |
| 사용자 정의 카테고리 | 사용자가 직접 추가한 카테고리 |
| 완료 (Completion) | 할일의 수행이 끝난 상태 (`isCompleted = true`) |
| 종료 예정일 (Due Date) | 할일을 완료해야 하는 목표 날짜 |
| 인증 (Authentication) | 이메일·비밀번호로 신원을 확인하고 JWT 토큰을 발급받는 행위 |
| Access Token | 인증된 API 요청에 사용하는 단기 JWT (만료: 1시간, Zustand 메모리 저장) |
| Refresh Token | Access Token 재발급에 사용하는 장기 토큰 (만료: 7일, Zustand 메모리 저장, 재발급 요청 시 바디로 전송) |

## 5. 경계 컨텍스트 (Bounded Context)

### 5.1 인증/사용자 관리 컨텍스트
- 책임: 회원가입, 로그인/로그아웃, JWT 토큰 발급·갱신, 사용자 정보 수정
- 핵심 엔티티: User
- 외부 의존: 없음
- 관련 BR: BR-3.1.1, BR-3.1.2, BR-3.1.3

### 5.2 할일 관리 컨텍스트
- 책임: 할일 CRUD, 완료 처리, 목록 필터링
- 핵심 엔티티: Todo
- 외부 의존: 인증 컨텍스트(userId 검증), 카테고리 컨텍스트(categoryId 검증)
- 관련 BR: BR-3.2.1 ~ BR-3.2.5

### 5.3 카테고리 관리 컨텍스트
- 책임: 기본 카테고리 제공(DB 시딩), 사용자 정의 카테고리 CRUD
- 핵심 엔티티: Category
- 외부 의존: 인증 컨텍스트(userId 검증)
- 관련 BR: BR-3.3.1, BR-3.3.2, BR-3.3.3

## 6. 주요 유스케이스

| # | 유스케이스 | 액터 | 간략 흐름 | 관련 비즈니스 규칙 |
|---|-----------|------|-----------|-------------------|
| UC-01 | 회원가입 | 비인증 사용자 | 이메일·비밀번호·이름 입력 → 유효성 검증 → 계정 생성 → JWT 발급 | BR-3.1.1, BR-3.1.2 |
| UC-02 | 로그인 | 비인증 사용자 | 이메일·비밀번호 입력 → 인증 → Access/Refresh Token 발급 | BR-3.1.2 |
| UC-03 | 사용자 정보 수정 | 인증 사용자 | 수정할 항목(이름 또는 비밀번호) 입력 → 저장 | BR-3.1.3 |
| UC-04 | 할일 등록 | 인증 사용자 | 제목·시작일·종료예정일·카테고리 입력(설명 선택) → 저장 | BR-3.2.1, BR-3.2.3, BR-3.2.4 |
| UC-05 | 할일 수정 | 인증 사용자 | 본인 할일 선택 → 항목 수정 → 저장 | BR-3.2.2, BR-3.2.4 |
| UC-06 | 할일 삭제 | 인증 사용자 | 본인 할일 선택 → 삭제 확인 → 삭제 | BR-3.2.2 |
| UC-07 | 할일 완료 처리 | 인증 사용자 | 할일 선택 → 완료↔미완료 상태 토글 | BR-3.2.2, BR-3.2.5 |
| UC-08 | 할일 목록 필터링 | 인증 사용자 | 필터 조건 선택(AND 조합) → 필터링된 목록 조회 | BR-3.2.2 |
| UC-09 | 카테고리 추가 | 인증 사용자 | 카테고리명 입력 → 사용자 정의 카테고리 생성 | BR-3.3.2 |
| UC-10 | 카테고리 삭제 | 인증 사용자 | 사용자 정의 카테고리 선택 → 연결 할일 없을 시 삭제 | BR-3.3.1, BR-3.3.3 |
| UC-11 | 카테고리 수정 | 인증 사용자 | 사용자 정의 카테고리 선택 → 카테고리명 수정 | BR-3.3.1, BR-3.3.2 |

### 6.1 유스케이스 시나리오

#### UC-01 회원가입

| 구분 | 시나리오 | 기대 응답 |
|------|---------|-----------|
| 성공 | 유효한 이메일·비밀번호(8자 이상)·이름 입력 | `201 Created` + Access/Refresh Token |
| 실패 | 이미 가입된 이메일 | `409 Conflict` |
| 실패 | 이메일 형식 오류 또는 비밀번호 8자 미만 | `400 Bad Request` |

#### UC-02 로그인

| 구분 | 시나리오 | 기대 응답 |
|------|---------|-----------|
| 성공 | 등록된 이메일·올바른 비밀번호 입력 | `200 OK` + Access/Refresh Token |
| 실패 | 이메일 미등록 또는 비밀번호 불일치 | `401 Unauthorized` |

#### UC-04 할일 등록

| 구분 | 시나리오 | 기대 응답 |
|------|---------|-----------|
| 성공 | 필수 항목 모두 입력, dueDate >= startDate | `201 Created` + 생성된 Todo |
| 실패 | dueDate < startDate | `400 Bad Request` |
| 실패 | 존재하지 않거나 타인 소유의 categoryId | `404 Not Found` |
| 실패 | 인증 토큰 없음 또는 만료 | `401 Unauthorized` |

#### UC-08 할일 목록 필터링

| 구분 | 시나리오 | 기대 응답 |
|------|---------|-----------|
| 성공 | 필터 없이 조회 | `200 OK` + 본인 할일 전체 목록 |
| 성공 | `categoryId` 필터 적용 | `200 OK` + 해당 카테고리 할일 목록 |
| 성공 | `isCompleted` 필터 적용 | `200 OK` + 완료/미완료 할일 목록 |
| 성공 | `dueDateFrom`·`dueDateTo` 기간 필터 적용 | `200 OK` + 기간 내 할일 목록 |
| 성공 | 복수 필터 동시 적용 (AND 조합) | `200 OK` + 조건 교집합 목록 |
| 실패 | 인증 토큰 없음 또는 만료 | `401 Unauthorized` |

> **필터 조합 방식:** 모든 필터 조건은 AND로 결합한다. 기간 필터의 기준 필드는 `dueDate`이며, `dueDateFrom` 이상 `dueDateTo` 이하인 항목을 반환한다.

#### UC-09 카테고리 추가

| 구분 | 시나리오 | 기대 응답 |
|------|---------|-----------|
| 성공 | 중복 없는 카테고리명 입력 | `201 Created` + 생성된 Category |
| 실패 | 동일 사용자 내 중복 카테고리명 | `409 Conflict` |
| 실패 | 카테고리명 50자 초과 | `400 Bad Request` |

#### UC-10 카테고리 삭제

| 구분 | 시나리오 | 기대 응답 |
|------|---------|-----------|
| 성공 | 연결된 할일이 없는 사용자 정의 카테고리 삭제 | `204 No Content` |
| 실패 | 연결된 할일이 존재하는 카테고리 삭제 시도 | `409 Conflict` |
| 실패 | 기본 카테고리 삭제 시도 | `403 Forbidden` |
| 실패 | 타인 소유 카테고리 삭제 시도 | `403 Forbidden` |

#### UC-11 카테고리 수정

| 구분 | 시나리오 | 기대 응답 |
|------|---------|-----------|
| 성공 | 중복 없는 새 카테고리명으로 수정 | `200 OK` + 수정된 Category |
| 실패 | 동일 사용자 내 중복 카테고리명으로 수정 시도 | `409 Conflict` |
| 실패 | 카테고리명 50자 초과 | `400 Bad Request` |
| 실패 | 기본 카테고리 수정 시도 | `403 Forbidden` |
| 실패 | 타인 소유 카테고리 수정 시도 | `403 Forbidden` |

## 1. 프로젝트 개요

| 항목 | 내용 |
|------|------|
| 프로젝트명 | TodolistApp |
| 목적 | 인증된 사용자 단위로 할일을 안전하게 저장·관리하는 개인 할일 관리 웹 애플리케이션 |
| 핵심 문제 | 직장인이 업무·개인 할일을 카테고리별로 분류하고, 어디서든 웹 브라우저로 접근하여 효율적으로 관리할 수단이 부족함 |
| 1차 출시 목표 | MVP 기능 구현 및 배포 (개발 기간 3일) |

### 2.3 사용자 니즈

- 로그인 후 빠르게 할일을 등록하고 완료 처리할 수 있어야 함
- 업무·개인·기타 카테고리로 할일을 분류하고 필터링할 수 있어야 함
- PC와 모바일에서 동일한 데이터를 끊김 없이 사용할 수 있어야 함

## 5. 비기능 요구사항

### 5.1 성능

| 항목 | 목표 |
|------|------|
| 동시 접속자 | 최대 300명 |
| API 응답 시간 | 일반 CRUD p95 < 300ms |
| 페이지 최초 로드 | 3G 환경 기준 LCP < 3초 |

### 5.2 보안

| 항목 | 정책 |
|------|------|
| 비밀번호 저장 | bcrypt 해시 (salt rounds ≥ 10) |
| 토큰 저장 | Access Token·Refresh Token 모두 Zustand 스토어(메모리)에만 저장 — `localStorage`/`sessionStorage`/Cookie 미사용 |
| Access Token | JWT, 만료 1시간, `Authorization: Bearer` 헤더로 전달 |
| Refresh Token | JWT, 만료 7일, 로그인 응답 바디로 수신 후 Zustand 메모리에 보관, 재발급 요청 시 바디로 전송 |
| 로그아웃 | Zustand 스토어 초기화 (메모리에서 두 토큰 즉시 삭제) — 서버 요청 불필요 |
| CORS | 허용 Origin 환경변수로 명시 (`*` 금지) — Cookie 미사용으로 `credentials: true` 불필요 |
| 인가 | 모든 리소스는 본인 소유 여부를 서버에서 검증 |
| HTTPS | 운영 환경 필수 적용 |
| 입력값 검증 | 서버 측 유효성 검증 필수 (클라이언트 검증은 UX 보조 역할) |

### 5.3 데이터 보존 정책

| 이벤트 | 처리 |
|--------|------|
| 회원 탈퇴 | 사용자 계정 및 연관 데이터(Todo, 사용자 정의 Category) 즉시 물리 삭제 |
| 할일 삭제 | 즉시 물리 삭제 |
| 카테고리 삭제 | 연결 할일 없을 시 즉시 물리 삭제 |

### 5.4 가용성 및 확장성

| 항목 | 내용 |
|------|------|
| 운영 규모 | 소규모 개인 프로젝트 |
| 확장 고려 | OAuth 인증 추가를 고려한 인증 모듈 설계 |

| 스타일링 | 미정 (TBD) |

### 6.2 백엔드

| 항목 | 기술 |
|------|------|
| 런타임 | Node.js (JavaScript) |
| 프레임워크 | Express |
| API 방식 | REST API |
| DB 연동 | **pg 라이브러리** (PostgreSQL 17 직접 연결, ORM 사용 금지) |
| 인증 | jsonwebtoken (JWT 서명·검증) |
| 비밀번호 | bcrypt |

> **pg 라이브러리 사용 필수:** PostgreSQL DB 연동은 Prisma, TypeORM 등 ORM을 사용하지 않고 반드시 `pg` 라이브러리로 직접 쿼리를 작성한다.

### 6.3 데이터베이스

| 항목 | 기술 |
|------|------|
| DBMS | PostgreSQL 17 |
| 초기 데이터 | 기본 카테고리 3종 (`업무`, `개인`, `기타`) DB 시딩 |

### 6.4 인증 확장 계획

```
1차: JWT (이메일/비밀번호)
       ↓
2차: OAuth 2.0 Social 로그인 추가 (Google, Facebook 등)
     → 인증 모듈은 Provider 추상화를 고려하여 설계
```

## 7. UX / UI 방향

### 7.1 1차 MVP UI 원칙

- **단순함 우선:** 할일 등록·완료·필터링을 3번 이내 탭/클릭으로 완료할 수 있어야 함
- **반응형 필수:** 모바일(320px~)부터 데스크탑(1920px)까지 레이아웃 대응
- **피드백 명확성:** 성공/실패 액션에 대해 즉각적인 시각 피드백 제공 (토스트, 인라인 에러 등)
- **로딩 상태:** API 요청 중 스켈레톤 또는 스피너로 로딩 상태 표시

### 7.2 주요 화면 목록

| 화면 | 설명 |
|------|------|
| 회원가입 | 이메일·비밀번호·이름 입력 폼 |
| 로그인 | 이메일·비밀번호 입력 폼 |
| 할일 목록 | 필터 바 + 할일 카드 리스트 (메인 화면) |
| 할일 등록/수정 | 모달 또는 사이드 패널 |
| 카테고리 관리 | 카테고리 목록 + 추가/수정/삭제 |
| 사용자 설정 | 이름·비밀번호 수정, 회원 탈퇴 |

## 8. API 설계 원칙

- **REST 컨벤션:** 리소스 중심 URL, HTTP 메서드로 행위 표현
- **응답 형식:** JSON, 일관된 에러 구조 사용
- **인증:** `Authorization: Bearer <accessToken>` 헤더
- **에러 코드:** HTTP 상태 코드 + 도메인 에러 메시지 (도메인 정의서 시나리오 기준)

### 8.1 예상 엔드포인트 목록

```
POST   /api/auth/register          # 회원가입
POST   /api/auth/login             # 로그인
POST   /api/auth/logout            # 로그아웃 (클라이언트 Zustand 스토어 초기화)
POST   /api/auth/refresh           # Access Token 재발급

GET    /api/users/me               # 내 정보 조회
PATCH  /api/users/me               # 내 정보 수정 (UC-03)
DELETE /api/users/me               # 회원 탈퇴 (계정·연관 데이터 즉시 삭제)

GET    /api/todos                  # 할일 목록 조회 (?categoryId=&isCompleted=&dueDateFrom=&dueDateTo=)
POST   /api/todos                  # 할일 등록 (UC-04)
GET    /api/todos/:id              # 할일 단건 조회
PATCH  /api/todos/:id              # 할일 수정 (UC-05) / 완료 처리 토글 (UC-07, isCompleted 포함)
DELETE /api/todos/:id              # 할일 삭제 (UC-06)

GET    /api/categories             # 카테고리 목록 조회
POST   /api/categories             # 카테고리 추가 (UC-09)
PATCH  /api/categories/:id         # 카테고리 수정 (UC-11, 사용자 정의 카테고리만 가능)
DELETE /api/categories/:id         # 카테고리 삭제 (UC-10)
```

> **UC-07 완료 처리:** 전용 엔드포인트 없이 `PATCH /api/todos/:id` 요청 바디에 `{ "isCompleted": true/false }` 를 포함하여 처리한다. 완료↔미완료 양방향 전환이 가능하다.

> **토큰 재발급:** `POST /api/auth/refresh` 요청 시 Zustand에 보관 중인 Refresh Token을 요청 바디 `{ "refreshToken": "..." }` 로 전달한다. Cookie를 사용하지 않는다.

## 1. 페르소나별 대표 시나리오

### 1.1 김민준 시나리오 (29세, 스타트업 개발자)

아침 8시 20분, 김민준은 지하철 2호선 강남역으로 향하는 열차 안에서 스마트폰을 꺼내 TodolistApp에 접속한다. 어제 퇴근하며 처리하지 못한 업무들이 머릿속에 맴돌아 불안했는데, 빠르게 등록해 두면 마음이 편해질 것 같았다.

민준은 이미 가입된 계정으로 로그인한다. 이메일과 비밀번호를 입력하자 곧바로 메인 할일 목록 화면으로 이동한다. 화면 상단의 카테고리 필터에서 '업무'를 선택하니 업무 관련 할일만 깔끔하게 정렬된다. 오늘 처리해야 할 항목들을 한눈에 확인한 뒤, 어젯밤 슬랙에서 받은 긴급 요청 사항을 새 할일로 등록한다. 제목에 "결제 모듈 버그 수정", 카테고리 '업무', 시작일과 종료예정일 모두 오늘 날짜로 입력하고 저장한다. 3번의 탭으로 등록이 완료된다.

퇴근 무렵 오후 6시, 민준은 PC 웹 브라우저에서 동일한 계정으로 접근한다. 아침에 모바일로 등록한 할일이 그대로 목록에 보인다. 결제 모듈 버그 수정이 완료되었으므로 해당 항목의 완료 토글을 클릭한다. 항목이 완료 상태로 전환되며 시각적으로 구분된다. '미완료' 필터를 적용해 아직 남은 업무를 확인하고 하루를 마무리한다.

### 1.2 이수진 시나리오 (38세, 마케팅 팀장)

이수진은 오전 9시에 회사 PC에 앉아 TodolistApp을 열고 로그인한다. 그녀는 여러 캠페인을 동시에 관리하고 있어 할일이 쌓이면 무엇부터 해야 할지 파악이 어렵다. 오늘은 먼저 종료예정일 기간 필터를 활용해 이번 주 안에 마감해야 할 항목들을 추려낸다. 필터 바에서 dueDateFrom을 오늘, dueDateTo를 이번 주 금요일로 설정하자 이번 주 마감 할일 목록만 표시된다.

목록을 살펴보던 수진은 '신제품 런칭 SNS 콘텐츠 기획' 할일의 종료예정일이 어제로 지나쳤음을 발견한다. 해당 항목을 클릭해 수정 화면을 열고, 종료예정일을 다음 주 화요일로 변경하고 저장한다. 이어서 카테고리 필터에서 '마케팅'을 선택하려 했지만, 해당 카테고리가 없음을 알게 된다. 카테고리 관리 화면으로 이동해 '마케팅'이라는 이름의 사용자 정의 카테고리를 추가한다. 이후 관련 할일들의 카테고리를 '마케팅'으로 수정한다.

점심 후 외부 미팅에서 대리점 담당자로부터 새로운 요청을 받은 수진은 스마트폰으로 TodolistApp에 접속해 '대리점 프로모션 제안서 작성' 할일을 빠르게 등록한다. 퇴근 전 PC로 돌아와 완료된 항목들을 완료 처리하고, 미완료 필터를 통해 내일의 할 일을 확인한 뒤 하루를 마친다.

### 1.3 박성호 시나리오 (51세, 중간 관리자)

박성호는 IT 도구에 익숙하지 않다. 팀원에게서 TodolistApp을 추천받아 오늘 처음 가입해 보기로 했다. 회원가입 화면에서 이메일 주소, 비밀번호, 이름만 입력하면 된다고 하니 간단해 보인다. 8자 이상의 비밀번호를 입력하고 가입 버튼을 누르자 바로 로그인 상태로 메인 화면이 열린다.

회의가 방금 끝났다. 후속 조치 사항이 세 가지다. 성호는 할일 등록 버튼을 누르고 첫 번째 항목을 입력한다. '2라인 생산량 점검', 카테고리는 기본 카테고리인 '업무'를 선택하고, 시작일과 종료예정일을 오늘로 설정한다. 복잡한 설명은 입력하지 않고 저장한다. 나머지 두 항목도 같은 방식으로 빠르게 등록한다.

오후에 '2라인 생산량 점검'을 완료했다. 목록에서 해당 항목을 찾아 완료 토글을 누른다. 항목이 완료 상태로 바뀐다. 화면이 복잡하지 않고 클릭 한 번으로 완료 처리가 되니 수첩보다 편리하다고 느낀다. 퇴근 즈음 로그아웃 버튼을 눌러 안전하게 세션을 종료한다.

## 2. UC별 상세 시나리오

### UC-01 회원가입

| 구분 | 내용 |
|------|------|
| 사전 조건 | 사용자가 아직 계정을 보유하지 않은 상태이다. 회원가입 화면에 접근 가능하다. |
| 정상 흐름 | 1. 사용자가 회원가입 화면에서 이메일, 비밀번호(8자 이상), 이름을 입력한다. 2. '가입하기' 버튼을 클릭한다. 3. 서버가 이메일 형식, 비밀번호 길이, 이메일 중복 여부를 검증한다. 4. 검증 통과 시 계정을 생성하고, Access Token(1시간)과 Refresh Token(7일)을 응답 바디로 발급한다. 5. 클라이언트가 두 토큰을 Zustand 스토어(메모리)에 저장한다. Cookie·localStorage는 사용하지 않는다. 6. 사용자가 메인 할일 목록 화면으로 이동한다. |
| 사후 조건 | 신규 사용자 계정이 DB에 생성된다. 사용자는 인증 상태로 메인 화면에 진입한다. |
| 예외 흐름 | - 이미 가입된 이메일 입력: 409 Conflict 오류 메시지("이미 사용 중인 이메일입니다")가 표시되며 가입이 중단된다. - 이메일 형식 오류: 400 Bad Request, 이메일 형식 안내 메시지가 인라인으로 표시된다. - 비밀번호 8자 미만: 400 Bad Request, 비밀번호 길이 안내 메시지가 인라인으로 표시된다. - 이름 미입력: 400 Bad Request, 필수 항목 안내 메시지가 표시된다. |

### UC-02 로그인

| 구분 | 내용 |
|------|------|
| 사전 조건 | 사용자가 이미 회원가입을 완료한 상태이다. 로그인 화면에 접근 가능하다. |
| 정상 흐름 | 1. 사용자가 로그인 화면에서 이메일과 비밀번호를 입력한다. 2. '로그인' 버튼을 클릭한다. 3. 서버가 이메일 존재 여부와 비밀번호(bcrypt 검증)를 확인한다. 4. 인증 성공 시 Access Token(1시간)과 Refresh Token(7일)을 응답 바디로 반환한다. 5. 클라이언트가 두 토큰을 Zustand 스토어(메모리)에 저장한다. 6. 사용자가 메인 할일 목록 화면으로 이동한다. |
| 사후 조건 | 사용자가 인증 상태로 메인 화면에 진입한다. Access Token과 Refresh Token이 Zustand 스토어(메모리)에 보관된다. |
| 예외 흐름 | - 등록되지 않은 이메일 입력: 401 Unauthorized, "이메일 또는 비밀번호가 올바르지 않습니다" 메시지가 표시된다. - 비밀번호 불일치: 401 Unauthorized, 동일한 보안 메시지가 표시된다(이메일 존재 여부를 노출하지 않음). - 이메일 또는 비밀번호 미입력: 400 Bad Request, 필수 항목 안내 메시지가 표시된다. |

### UC-04 할일 등록

| 구분 | 내용 |
|------|------|
| 사전 조건 | 사용자가 로그인된 인증 상태이다. 최소 하나의 카테고리(기본 카테고리 포함)가 존재한다. |
| 정상 흐름 | 1. 사용자가 메인 화면에서 할일 등록 버튼을 클릭한다. 2. 등록 모달(또는 사이드 패널)이 열린다. 3. 사용자가 제목(필수), 시작일(필수), 종료예정일(필수), 카테고리(필수)를 입력하고 설명(선택)을 입력한다. 4. '저장' 버튼을 클릭한다. 5. 서버가 인증 여부, 입력값 유효성, dueDate >= startDate 조건, 카테고리 소유권을 검증한다. 6. 검증 통과 시 할일 레코드를 생성하고 201 Created와 생성된 Todo를 반환한다. 7. 할일 목록이 갱신되어 새 항목이 표시된다. |

| 예외 흐름 | - 종료예정일이 시작일보다 이전: 400 Bad Request, "종료예정일은 시작일 이후여야 합니다" 안내 메시지가 표시된다. - 카테고리 미선택: 400 Bad Request, 카테고리 필수 입력 안내 메시지가 표시된다. - 존재하지 않거나 타인 소유의 카테고리 ID 사용: 404 Not Found 오류가 반환된다. - 제목 200자 초과: 400 Bad Request, 제목 길이 제한 안내 메시지가 표시된다. - 설명 2000자 초과: 400 Bad Request, 설명 길이 제한 안내 메시지가 표시된다. - 인증 토큰 없음 또는 만료: 401 Unauthorized, 로그인 화면으로 이동한다. |

| 구분 | 내용 |

| 정상 흐름 | 1. 사용자가 할일 목록에서 삭제하고자 하는 항목의 삭제 버튼을 클릭한다. 2. 삭제 확인 다이얼로그가 표시된다("정말 삭제하시겠습니까?"). 3. 사용자가 '확인' 버튼을 클릭한다. 4. 서버가 해당 할일의 소유자가 요청한 사용자인지 검증한다(BR-3.2.2). 5. 검증 통과 시 해당 Todo 레코드를 DB에서 즉시 물리 삭제한다. 6. 204 No Content를 반환하고 할일 목록에서 해당 항목이 제거된다. |

### UC-07 완료 처리 토글

| 구분 | 내용 |
|------|------|
| 사전 조건 | 사용자가 로그인된 인증 상태이다. 본인 소유의 할일이 하나 이상 존재한다. |
| 정상 흐름 (미완료 → 완료) | 1. 사용자가 할일 목록에서 미완료 상태인 항목의 완료 토글을 클릭한다. 2. 클라이언트가 PATCH /api/todos/:id 요청에 { "isCompleted": true }를 전송한다. 3. 서버가 해당 할일의 소유자가 요청한 사용자인지 검증한다(BR-3.2.2). 4. isCompleted를 true로 갱신하고 200 OK와 수정된 Todo를 반환한다. 5. 할일 항목이 완료 상태 스타일로 변경된다(예: 취소선, 색상 변경). |
| 정상 흐름 (완료 → 미완료) | 1. 사용자가 완료 상태인 항목의 완료 토글을 다시 클릭한다. 2. 클라이언트가 PATCH /api/todos/:id 요청에 { "isCompleted": false }를 전송한다. 3. 서버가 소유자를 검증한 후 isCompleted를 false로 갱신한다. 4. 할일 항목이 미완료 상태 스타일로 복원된다. |
| 사후 조건 | 해당 Todo 레코드의 isCompleted 값이 변경된다. 완료 처리된 데이터는 삭제 전까지 보존된다(BR-3.2.5). |
| 예외 흐름 | - 타인 소유의 할일 토글 시도: 403 Forbidden 오류가 반환된다. - 존재하지 않는 할일 ID: 404 Not Found 오류가 반환된다. - 인증 토큰 없음 또는 만료: 401 Unauthorized, 로그인 화면으로 이동한다. |

### Access Token 재발급

| 구분 | 내용 |
|------|------|
| 사전 조건 | 사용자의 Access Token이 만료되었으나 Zustand 스토어의 Refresh Token이 아직 유효하다. |
| 정상 흐름 | 1. 클라이언트가 API 요청 중 401 Unauthorized 응답을 받는다(Access Token 만료). 2. 클라이언트 인터셉터가 Zustand에서 Refresh Token을 꺼내 POST /api/auth/refresh 요청 바디에 담아 전송한다. 3. 서버가 요청 바디의 Refresh Token 유효성과 만료 여부를 검증한다. 4. 검증 성공 시 새 Access Token(1시간)을 발급하여 응답 바디로 반환한다. 5. 클라이언트가 새 Access Token을 Zustand에 저장하고 실패했던 원래 API 요청을 자동 재시도한다. 6. 사용자는 중단 없이 서비스를 계속 이용한다. |
| 사후 조건 | 새 Access Token이 발급되고 Zustand 스토어(메모리)에 저장된다. 사용자는 별도의 로그인 과정 없이 인증 상태를 유지한다. |
| 예외 흐름 | - Refresh Token도 만료된 경우: 401 Unauthorized, 클라이언트가 Zustand 스토어를 초기화하고 로그인 화면으로 이동한다. "세션이 만료되었습니다. 다시 로그인해 주세요" 메시지가 표시된다. - Refresh Token이 위변조된 경우: 401 Unauthorized, 동일하게 Zustand 초기화 후 로그인 화면으로 이동한다. - 페이지 새로고침: Zustand 메모리가 초기화되어 두 토큰이 사라지므로 재로그인이 필요하다. - 네트워크 오류로 재발급 요청 실패: 재시도 후 계속 실패 시 로그인 화면으로 이동한다. |

### 회원 탈퇴

| 구분 | 내용 |
|------|------|
| 사전 조건 | 사용자가 로그인된 인증 상태이다. 사용자 설정 화면에 접근 가능하다. |

| 사후 조건 | 사용자의 모든 데이터(User, Todo, 사용자 정의 Category)가 DB에서 완전히 삭제된다. 기본 카테고리(업무, 개인, 기타)는 삭제되지 않는다. 사용자는 비인증 상태로 전환되며 동일 이메일로 재가입이 가능하다. |
| 예외 흐름 | - 탈퇴 확인 다이얼로그에서 '취소' 클릭: 탈퇴가 취소되고 사용자 설정 화면으로 복귀한다. - Access Token 만료 상태에서 요청: 자동 토큰 재발급 후 탈퇴 요청을 재시도한다. Refresh Token도 만료된 경우 로그인 화면으로 이동한다. - 서버 오류로 삭제 처리 실패: 500 Internal Server Error, "탈퇴 처리 중 오류가 발생했습니다. 잠시 후 다시 시도해 주세요" 메시지가 표시되며 데이터는 삭제되지 않고 보존된다. |
